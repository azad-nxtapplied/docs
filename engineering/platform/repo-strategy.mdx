## 1. The decision: evolve the repo — it's not refactor vs. scratch, it's old body, new organs

We build on the existing repository. The expensive, slow-to-rebuild assets are exactly what it already has in production: the hash-chained audit ledger, workspace tenancy woven through every model, policy gates with idempotent auto-execution, the inbox→case→communication spine live at the first customer, 243 disciplined migrations, CI→Helm→k3s with observability. Roughly 70% of the kernel guarantees are real today. A greenfield rewrite would spend its first year reproducing them while the first deployments wait — and it would freeze the evidence: production audit chains, case history, and learning signals are platform IP under our own positioning; scratch resets them to zero.

At the same time, most of what the v4.0 map demands is new by definition — Evaluation Engine, Learning Engine, Skill Runtime, the knowledge half of AI Memory, Service Contract, Metering. Those are built greenfield *inside* the repo, against the new package boundaries, with enforcement on from day one. The honest composition:

- **~50% new organs:** built clean in the new structure.

- **~30% re-homed:** existing production code moved behind new names and boundaries — renaming and boundary-drawing, not rewriting.

- **~15% quarantined:** the AP estate into the Founders Finance module; payment execution hard-flagged, internal-only.

- **~5% deleted:** dead experiments. Deletion is a gift; a funeral list is part of the first sprint.

**The contamination rule (the one that decides whether this works):** no new file lands outside the new package structure — ever, including hotfixes. Enforced by the import-linter ratchet: the legacy allow-list may only shrink; CI fails on growth.

## 3. Modularization: contracts, not processes

**Code modules — hard.** One package per kernel element with an explicit public interface: kernel.gateway · skill_runtime · memory · auditor · evaluation · learning · policy · autonomy · integration · core, then suites.caseops, suites.embedded, modules.*, deployments.*. Other packages import only a module's public surface, never internals — the same import-linter that enforces layers enforces module interfaces. This is what makes Memo 12's substrates swappable (LiteLLM→Bifrost, Langfuse→Phoenix) without distributed-systems tax.

**Deployables — conservative.** Exactly four of ours, each with a physical reason: **api** (request surface), **worker** (CPU-bound and batch), **skill-runtime** (the AgentCore successor: different scaling and timeouts, and a blast-radius wall between code-that-talks-to-models and code-that-holds-the-ledger), **telemetry-exporter** (small, security-critical, and politically separate — one tiny auditable binary answers "what leaves our perimeter"). Adopted deployables beside them: gateway proxy, Keycloak, Langfuse. Anyone proposing a fifth NXT deployable owes a physical reason.

## 4. Language stack

**Python 3.12+ for everything with semantics** — kernel, suites, skills, playbook tooling. Strictly typed (mypy in CI), Pydantic as the single schema source of truth. Rationale: team gravity (the engineers, the 117k existing LoC, the strangler interop are Python), ecosystem gravity (the entire adopted AI substrate is Python-first), and the fact that the one genuine Go performance case — the gateway hot path — is *bought*, not written (Memo 12).

**TypeScript for the surfaces and the first SDK**, generated from OpenAPI so the contract has one origin. **Go permitted only for contract-speaking sidecars** with zero business logic and a real distribution case — telemetry exporter first, customer-ops CLI if ever. **The "Go for the platform layer" intention is formally retired.** Escape hatch on record: a profiled, *measured* Python hotspot in the kernel may become a Rust/Go extension behind the same module interface — a weekend, not a stack decision.

## 5. Performance & concurrency doctrine

**SLOs per interaction class:** interactive reads **≤300ms p95** (served from state) · live skill calls **≤3s p95**, streaming · batch unbounded, throughput-monitored.

**The budget truth:** on a live AI call, the model costs 500–3,000ms; our entire orchestration, policy, and recording overhead is 5–25ms. Language optimizes ~1% of the budget; architecture optimizes the rest. Per-instance BYO means every deployment serves one customer, not the internet: load is bounded and forecastable, and the scaling lever is replicas.

**Design rules (hard requirements, not preferences):**

- 1\. **Serve-from-state, refresh-behind** wherever the product allows: for the ARZ copilot, suggestions are computed asynchronously by the Skill Runtime when a case opens or changes — fully governed, fully recorded, off the clock — and the copilot endpoint *reads* that state in \<50ms with explanation, confidence, and evidence links attached. The fast path and the attestable path are the same path. Live calls use fast model tiers, streaming, and context caching.

- 2\. **Nothing synchronous on the hot path except the decision itself.** Audit records, metering events, and learning signals write to ordered outboxes; consumers persist them asynchronously.

- 3\. **The chain sequencer:** the hash chain is per-workspace single-writer by nature; therefore chaining happens off the hot path — outbox-ordered, one sequencer per workspace — giving guaranteed order with zero hot-path contention. Verification runs on the sealed chain.

- 4\. **Never hold a DB connection across a model call.** Load context, release, await, reacquire to record. Lintable; linted.

- 5\. **CPU work lives only in workers**, in queue classes (interactive/standard/batch) with separate pools; ingestion surges can never queue ahead of a copilot refresh. Backpressure degrades to *last computed state with a staleness flag* — degraded freshness beats degraded latency, and the platform records the degradation.

- 6\. **Idempotency-first atomicity:** key reservation is the first atomic act of any service-contract call; concurrent retries collapse before money is spent.

- 7\. **Acceptance gate:** a load test at **5× projected ARZ peak** on our infrastructure passes before integration testing with the partner begins. Vendor benchmarks don't count.

- 8\. **The known ceiling, pre-answered:** gateway-proxy tail latency under sustained concurrency is the documented Python weak point; the Bifrost swap trigger (Memo 12) is the bought escape hatch, contained behind our gateway interface.

## 6. Pattern-B tenancy: instance → workspace → claims

The ARZ deployment runs one central service used by many of the partner's customers, each with their own software instance and users. The structure:

**Instance = the partner (ARZ/Teemer).** One NXT Core deployment in their environment; perimeter, contract, operating annex at this level.

**Workspace = one partner customer** (one practice / one partner software instance). Hard isolation on every record, case, and piece of evidence — structurally provable. **New requirement: workspace provisioning becomes an Admin-API operation** — partner onboards a customer, calls the API, workspace exists with credentials.

**Identity: machines authenticate; end users are claims.** Practice users never become NXT users — they live in the partner's software, authenticated there. Each partner software instance holds machine credentials **bound to exactly its workspace** (a call with practice A's credentials physically cannot reach practice B — enforced at credential level). The human travels as a claim: every service-contract call carries on_behalf_of: \<partner user reference\>, recorded in the Execution's actor — so the audit trail still answers "which person at which practice triggered and accepted this," while NXT manages zero end-user accounts and minimal personal data. Humans who log into NXT surfaces are partner staff only (partner ops view, role-scoped).

**Four explicit designs on top:**

- 1\. **Shared knowledge, isolated cases:** billing-rule knowledge, skills, and the playbook are instance-level assets entitled to all workspaces — one knowledge version, evaluated once, serving every practice; cases and evidence remain workspace-isolated.

- 2\. **Cross-workspace learning is governed, contractual, and curated:** no raw case content crosses workspaces; curated, de-identified dataset candidates flow to playbook-level improvement through the Learning Engine's gates. A contract clause, not just a code path — and a selling point for the partner: every customer makes the component better for all.

- 3\. **Metering per workspace is the partner's downstream invoice:** our per-workspace usage counters become the meter the partner bills their customers with — surfaced in the partner ops view.

- 4\. **Fairness inside the instance:** per-workspace rate budgets on the §5 queue classes; one chain's batch run cannot starve a small practice's copilot.

**Visibility boundary, stated up front:** the partner ops view shows aggregate quality and per-workspace operational health by default; per-workspace case *content* visibility is an entitlement the partner's contracts with their customers must cover — configurable per workspace, off by default. Their data-protection homework, our switch.

**Net-new build items from this section** (all on the ARZ critical path anyway): workspace provisioning API, credential-to-workspace binding, per-workspace budgets, the on-behalf-of claim convention. ~2 engineer-weeks total.
