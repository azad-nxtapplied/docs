NXT Core is an AI-native platform that runs inside the customer's environment. It is organized into four **bands** — from the customer-facing playbooks at the top down to the AI kernel — all resting on a shared **Foundation** (Platform Core). The bands hold the AI-specific machinery; the Foundation is the generic plumbing that every band uses. Each band builds on the bands below it, and everything builds on the Foundation.

**Hover any service** in the diagram for a quick summary. The **foundation components** (Tiers 0–2, marked ↗) open their own contract page. Full written descriptions follow below the diagram.

## Components

What each service does and the responsibilities it owns.

### Band 4 · Deployment

| Component | Responsibility |
| --- | --- |
| **Configuration** | Holds each customer's workspace settings and secret bindings — connector credentials, taxonomy, and automation seeds. Pure data, synced from a secret manager. |
| **Customer Components** | The customer-owned systems the platform connects to (mailboxes, ERPs, identity providers). The platform reaches them through the Integration Fabric. |

### Band 3 · AI-native playbooks

Declarative configuration on top of the kernel: each playbook is an intent taxonomy, autonomy ceilings (L0–L3), risk hardstops, and a skill list.

| Component | Responsibility |
| --- | --- |
| **Customer Servicing** | The first playbook: handles complaints and service requests with graduated autonomy. |
| **Claim Management** | Handles claims intake and resolution on the shared Case Operations services and the kernel. |
| **Founder Finance** | Runs founder finance operations on the same primitives. |
| **Embedded AI Components** | Reusable AI components exposed via Service Contract and SDKs. Planned for a future release. |

### Band 2 · AI suites

| Component | Responsibility |
| --- | --- |
| **NXT Workspace** | The operator surface — inbox, case detail, and gate-review screens. A React app over the Platform API. |
| **NXT Cockpit** | Monitoring and metrics — throughput, autonomy mix, and escalation rate. Reads the audit and evaluation streams. |
| **NXT Admin** | Configuration and governance — manages autonomy ceilings and approves autonomy promotions. Gated by RBAC. |
| **Platform API** | The single typed boundary the Suites call. A REST surface over Band-2 services; every call carries tenant and correlation context. |
| **Agent Case Framework** | Owns the generic case lifecycle and park/resume protocol, and is the single writer of case status. |
| **Case Domain Memory** | Holds the case-scoped record — classification, entities, signals, comms thread, parked state. A case-shaped view over AI Memory. |
| **Case Execution Library** | A catalog of deterministic, idempotent, audited executors a case invokes once authorized (send email, route, close). |

### Band 1 · Kernel

**Execution plane**

| Component | Responsibility |
| --- | --- |
| **Model Gateway** | The single entry point to every LLM (Commercial, Open-Source, BYOM) behind one unified API. Applies enablement, guardrails, smart caching, execution with retry/fallback/circuit-breaker, and cost accounting. |
| **Skill Runtime** | Executes a single skill and emits typed artifacts. Runs are idempotent and durable — they park where the Autonomy verdict directs and resume on a human signal. |
| **Integration Fabric** | Handles all I/O with the outside world — inbound email and webhooks, outbound sends, and the tool transport for skills. Idempotent at the message boundary. |

**Governance plane**

| Component | Responsibility |
| --- | --- |
| **Policy Gate** | Runs first. Returns a deterministic ALLOW / DENY / REQUIRE_APPROVAL for any action by evaluating versioned policy-as-code as a pure function. |
| **Autonomy Controller** | Runs second. Calibrates how much autonomy an action gets now — `min(earned_level, policy_ceiling)` with risk hardstops — reading earned trust from the Learning Engine. |

**Record plane**

| Component | Responsibility |
| --- | --- |
| **AI Memory** | Operational recall in two tiers — Episodic (run/case history) and Knowledge/Grounding (RAG). Postgres + pgvector, hot cache in Redis. |
| **AI Auditor** | The immutable record of why every decision happened — append-only, hash-chained audit events on the event bus. |

**Feedback plane**

| Component | Responsibility |
| --- | --- |
| **Evaluation Engine** | Grades decision quality from explicit signals (override, edit, reopen) and implicit signals mined from the audit stream. |
| **Learning Engine** | Turns track record into earned trust — `earned_level` and eligibility, with operator-gated promotion. Feeds the Autonomy Controller. |

### Foundation · Platform Core

The layered foundation. Each component below has its own contract page.

| Tier | Component | Responsibility |
| --- | --- | --- |
| 2 | **[IAM (Identity & Access Management)](/engineering/architecture/components/iam)** | Identity, tenancy, and coarse authorization in one bounded context — who, where, and what they may do. |
| 1 | **[Configuration & Secrets](/engineering/architecture/components/configuration-secrets)** | Per-workspace config, feature flags, and secret bindings, synced from a secret manager. |
| 1 | **[Eventing](/engineering/architecture/components/eventing)** | The internal event backbone — publish primitives with a transactional outbox. |
| 0 | **[Data Adapter](/engineering/architecture/components/data-adapter)** | The shared, safe way to reach the DB (system of record) and cache (ephemeral). |
| — | **[Observability](/engineering/architecture/components/observability)** | Structured logging, metrics, and traces for system health (cross-cutting). |

## Design principles

<AccordionGroup>
  <Accordion title="Build downward" icon="arrow-down">
    Each tier builds only on the tiers below it, forming an acyclic dependency graph. Lower components stay mechanical — they read the `workspace_id` carried on each request, which keeps the layering clean.
  </Accordion>
  <Accordion title="Group by bounded context" icon="object-group">
    Related concerns live together: Identity, Tenancy, and Authorization form IAM; Config and Secrets belong together; Observability is cross-cutting. Industry-standard groupings.
  </Accordion>
  <Accordion title="Tenant isolation everywhere" icon="shield">
    Every request is scoped to its workspace by an app-level filter, backed by database row-level security, with the cache isolated by a workspace key prefix.
  </Accordion>
  <Accordion title="Types flow up" icon="arrow-up">
    A thin shared kernel holds universal primitives (IDs, base Event). Each component owns and exports its own contract DTOs; subtypes resolve via a `type` discriminator on the wire (CloudEvents-style).
  </Accordion>
  <Accordion title="Two complementary gates" icon="key">
    IAM gates _capability_ (do you have a licence). The Policy Gate gates the _specific action_ at execution time. Together they cover both questions.
  </Accordion>
</AccordionGroup>
