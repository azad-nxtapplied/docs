## 1. System context

One **instance** = one customer environment = the whole stack below, reproducible from configuration, inside the customer's perimeter. Multi-tenancy (workspaces) exists *inside* an instance; customers are isolated at instance level.

*\[System context diagram — see the digital version of this memo for the full schematic.\]*

Surfaces: **WS** = NXT Workspace, **CP** = NXT Cockpit, **AD** = NXT Admin (one SPA, role-gated bundles). Adopted deployables (Keycloak, LLM proxy, Langfuse) per Memo 12 sit beside NXT's four.

## 2. Deployables

Four NXT deployables, each with a physical reason (Memo 13 §3); everything else is in-process or adopted.

| **Deployable** | **Tech** | **Responsibility** | **Why separate** |
|----|----|----|----|
| app/api | Python 3.12, FastAPI, SQLAlchemy 2, Pydantic v2 | The Platform API /v1: synchronous reads/writes, policy evaluation, service-contract endpoints, webhooks in. Hosts the in-process kernel (Auditor, Policy Gate, Autonomy Controller, AI Memory, Integration Fabric, Platform Core) | Stateless request surface; scales on request load |
| app/worker | Celery on Redis | Async and CPU-bound work: ingestion, OCR, embedding, eval runs, learning curation, scheduled scans, outbox consumers | CPU isolation from the event loop; queue-class scaling |
| app/skill_runtime | Python (FastAPI internal) | Skill execution: bounded multi-step skills under autonomy envelopes; the AgentCore successor | Different scaling/timeouts; blast-radius wall between model-facing code and the ledger |
| app/exporter | Python (Go permitted later) | Telemetry export: the allow-listed quality-signal stream out of the perimeter | Small, security-critical, politically separate — one auditable thing crosses the boundary |
| app/model_serving | BentoML/KServe-class (adopted) | Serves NXT-trained models (rankers, classifiers) behind Model Access | Different scaling/GPU profile; adopted, not built |

In-process kernel modules (not deployables): Evaluation Engine and Learning Engine run primarily as worker jobs; Model Access is a client to the adopted LLM proxy and to the model-serving runtime; everything else lives in app/api and is shared by app/worker and app/skill_runtime as a library.

## 3. Repository structure & enforcement

Monorepo; the v4.0 component map is the package layout (Memo 13 §1–§3). New code lands only here; the import ratchet may only shrink.

```text
nxtcore/
  kernel/
    gateway/  skill_runtime/  memory/  auditor/  evaluation/
    learning/ policy/  autonomy/  integration/  core/
    (each exposes public.py — the only importable surface)
  suites/
    caseops/  embedded/
  modules/
    complaint_banking/  founders_finance/   (each: manifest.yaml, ip_class)
  deployments/
    arz/  swk/                               (each: manifest.yaml, ip_class)
  apps/
    api/  worker/  skill_runtime/  exporter/
  web/    (NXT Workspace · Cockpit · Admin — one SPA)
  sdk/    (generated TS, then Python)
  infra/  (helm, flyway, grafana|perses, dashboards)
```

**Enforced in CI:** import-linter contracts — a package imports only public.py of others; layers depend only downward; no module imports a sibling module; no code imports a model-provider SDK (only kernel.gateway). Legacy violations sit on an allow-list that may only shrink (ratchet check). manifest.yaml with ip_class required for every modules/ *and deployments/*; CI fails if missing.

## 4. Language & stack (Memo 13 §4)

Python 3.12+ for everything with semantics — kernel, suites, skills, playbook tooling — strictly typed (mypy in CI), Pydantic as the single schema source of truth. TypeScript for the SPA and the first SDK, generated from OpenAPI. Go permitted only for contract-speaking sidecars with zero business logic (the exporter is the first candidate; the LLM proxy is *adopted* Go, not written). Escape hatch: a measured kernel hotspot may become a Rust/Go extension behind the same module interface. SQL/Flyway is the data law.

## 5. Data architecture

### 5.1 One PostgreSQL per instance, schema-per-concern

- kernel: cases, work_items, exceptions, approvals, executions, evidence_links, policies, policy_versions, autonomy_states, outbox

- evals: datasets, dataset_items, eval_runs, eval_results, release_gates

- knowledge: collections, documents, chunks (pgvector), versions

- ops: metering, telemetry rollups, connector health, cost rollups

- mod\_\<module\> / dep\_\<deployment\>: domain tables — foreign keys may point **down** into kernel, never sideways or up (the import rules, mirrored at the data layer).

All business tables carry workspace_id with row scoping enforced in the repository layer (no unscoped query path exported). executions is append-only: the app role has INSERT/SELECT only, no UPDATE/DELETE. Migrations are Flyway, expand-contract only, so instances upgrade independently.

### 5.2 The Execution ledger and the outbox/chain-sequencer (Memo 13 §5)

```sql
executions (
  id uuid pk, workspace_id, case_id, work_item_id null,
  actor jsonb,             -- {type: human|agent|system, id, on_behalf_of}
  kind text,               -- classify | extract | draft | gate_eval | model_call | ml_inference
                           -- | context_assembly | send | post | approve | override | tool_call ...
  input_ref / output_ref,  -- evidence pointers, never inline payloads
  policy_eval_ref null, context_ref null, cost jsonb null,
  seq bigint null,         -- assigned by the per-workspace sequencer
  prev_hash bytea null, hash bytea null,   -- filled off the hot path
  created_at timestamptz
)
```

**Hot path** writes the execution row and an outbox entry — no hashing, no lock. **Off the hot path**, one **per-workspace sequencer** (a single-writer worker consuming the workspace's outbox in order) assigns seq and computes the hash chain (hash = H(canonical(row) \|\| prev_hash)). A scheduled verifier re-walks sealed chains. This makes the audit guarantee scale under concurrent skill steps: ordering is guaranteed, hot-path contention is zero, verification always runs on the sealed chain.

**Invariant:** every state transition and side effect goes through record_execution() — the single write path. Connection discipline: no DB connection is held across a model call (load context → release → await model → reacquire to record).

### 5.3 The four data planes in storage terms

Operational and Evidence are NXT-authoritative (Postgres + customer S3 object store, content-addressed, provenance-tagged, append-only evidence). Knowledge is versioned in knowledge.\*, activated only after eval gates. Reference is read-mostly mirrors in kernel.integration cache tables carrying source, fetched_at, freshness_ttl; reads past TTL raise Exception(stale_reference); the fetched_at of data used in a decision is captured in that decision's context_ref.

## 6. Contracts

### 6.1 Platform API — the single external surface

FastAPI, OpenAPI 3.1, URI-versioned /v1; generated TS/Python SDKs. The SPA consumes only /v1 (CI-checked). Internal/platform routes live on a separate, internally-authenticated surface, disabled in customer builds except the exporter. Resource groups: cases, work-items, approvals, exceptions, policies, autonomy, evals, knowledge, documents, communications, services (embedded), admin.

### 6.2 Embedded service contract (Pattern B / Way 2)

```http
POST /v1/services/{service}/requests          # idempotency-key required
  → 202 { request_id, case_id }

GET  /v1/services/{service}/requests/{id}
  → { status, result, explanation, confidence, evidence_url, execution_id }

POST <customer-callback>                       # HMAC-signed webhook on completion
```

Every request creates a Case and produces Executions; result is schema-versioned per service; explanation is stored as Evidence; evidence_url deep-links the audit journey; metering counts at request level. Asynchronous-first; a synchronous convenience wrapper with timeout ships in the SDK. Idempotency-key reservation is the first atomic act, so concurrent retries collapse before a model is called.

### 6.3 Connector contract (Integration Fabric)

Python protocols per tier: IngestionConnector; ReferenceConnector.get(entity, key) -\> Versioned\[T\] with declared freshness_ttl; HandoffConnector.execute(handoff: Idempotent\[T\]) -\> Receipt. All calls run inside record_execution(); failures map to typed Exceptions (connector_unavailable, stale_reference, handoff_rejected). Credentials per deployment from the secrets store, never in playbooks or code.

### 6.4 Skill Runtime contract

The runtime executes a skill plan (bounded steps; tool calls, LLM calls, and ML-model inferences) under an autonomy envelope `{max_tier, allowed_tools, spend_caps}` computed by the Autonomy Controller and enforced pre-call; it cannot be exceeded. Context is requested from kernel.memory.assemble(task, entitlements) returning content plus a signed manifest stored as context_ref. Every step is an Execution. Tools follow MCP definitions where applicable. The execution ladder holds: Model Access = one call; Skill Runtime = one skill; Agent Case Framework = one case (Way 2 stops at the runtime). A skill step targets either backend through Model Access; both are recorded identically.

## 7. Runtime flows

**Way 1 — a case.** Ingest (email→dedupe→classify, confidence-thresholded) → Case created → Agent Case Framework advances work items → a skill runs on the Skill Runtime (context from AI Memory, call via Model Access) → Policy Gate before any side effect → human checkpoint in NXT Workspace (Approval, immutable) → Integration Fabric handoff (idempotent) → analytics and learning signals emitted. Every step → outbox → sequencer → ledger.

**Way 2 — an embedded component, served from state (Memo 13 §5).** Customer software POSTs to the service endpoint → Case created headless → on case open/change the Skill Runtime computes suggestions **asynchronously** (extraction → completeness assessment against the knowledge collection → explanation), recorded and evaluated, stored as case proposal state → the interactive read endpoint returns that state in \<50ms with explanation/confidence/evidence → webhook on material change → sampled cases surface in the partner ops view → corrections feed the Learning Engine.

**Eval-gated change.** A proposed change (prompt vN+1, policy edit, knowledge version, playbook bump) → CI runs release gates against golden sets (Langfuse-substrate runner) → green → published artifact → per-environment, recorded promotion → live quality monitored; regression proposes demotion.

**Autonomy promotion.** Accumulated eval + live evidence → Autonomy Controller checks the playbook's release_gate/policy thresholds → promotion proposed → approved (human, in NXT Cockpit) → tier raised; a hard-block miss or drift alarm auto-proposes demotion.

## 8. Pattern-B tenancy (Memo 13 §6)

Instance = the partner (ARZ/Teemer). Workspace = one partner customer (provisioned via the Admin API, scoped machine credentials bound to exactly that workspace at the auth layer — a credential cannot address another workspace). End users are not NXT users: they travel as on_behalf_of claims recorded in the Execution actor. Knowledge/skills/playbook are instance-level assets entitled to all workspaces (one version, evaluated once); cases and evidence are workspace-isolated. Cross-workspace learning moves only curated, de-identified dataset candidates through the Learning Engine's gates — never raw case content. Metering is per workspace (the partner's downstream invoice basis). Per-workspace rate budgets on the queue classes provide noisy-neighbour fairness.

## 9. Performance & concurrency (Memo 13 §5)

SLOs: interactive reads ≤300ms p95 (served from state) · live skill calls ≤3s p95, streaming · batch unbounded, throughput-monitored. Design rules, all enforced: nothing synchronous on the hot path except the decision itself (audit, metering, learning write-behind via outboxes); serve-from-state + refresh-behind for interactive components; no DB connection held across a model call; CPU work only in workers, in queue classes (interactive/standard/batch) with backpressure degrading to last-computed state with a staleness flag; idempotency-first atomicity. Acceptance gate: a load test at 5× projected ARZ peak on our infrastructure passes before partner integration testing — our numbers, not vendor benchmarks. Known ceiling: gateway-proxy tail latency under sustained concurrency → the Bifrost swap trigger (Memo 12), contained behind kernel.gateway.

## 10. Security & deployment

**AuthN:** Keycloak per instance; humans via customer IdP federation (Entra ID); services and Pattern-B partners via scoped machine credentials / signed keys, workspace-bound. **AuthZ:** RBAC (operator, reviewer, approver, admin, auditor, partner) at the API; SoD in policy packs. **Tenancy:** workspace_id scoping in every repository call; cross-workspace access only via the platform role, disabled in customer instances. **Secrets:** External Secrets + SOPS; OpenBao if a manager is required (not Vault — BUSL). **Telemetry export:** dedicated app/exporter, schema-versioned allow-list (metrics, eval scores, trace metadata without payloads, error signatures), outbound-only, mutually authenticated, customer-inspectable — the only data leaving the perimeter besides governed handoffs to the customer's own systems. **Supply chain:** signed images, SBOM per release, license scan in CI (Memo 12 gate). **Deployment:** Kubernetes/Helm, k3d dev → staging → customer instances (BYO compute); one versioned release set (images + Helm chart + Flyway migrations + playbook bundles); per-environment, eval-gated, recorded promotion under the operating annex. Object storage defaults to the customer's S3-compatible store; bundled fallback is SeaweedFS (not MinIO — AGPL). Dashboards: Grafana where acceptable, Perses for license-strict customers.
