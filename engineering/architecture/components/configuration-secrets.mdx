Configuration & Secrets is the single place a component asks for a setting or a credential. It is a thin **resolver**: it does not own most of what it serves — it knows the right source for each value and presents one interface, always scoped to the current workspace, so a component never has to know where a value lives or how it is protected.

## Where each value comes from

| Value | Source of truth | How the platform reads it |
| --- | --- | --- |
| **Platform / static config** | Deployment config (env, ConfigMaps, Helm values) | Loaded at boot |
| **Per-workspace settings & feature flags** | The platform itself — edited in NXT Admin, versioned, exportable | Stored in the platform DB, cached for fast reads |
| **Secrets** (connector credentials, signing & encryption keys) | The customer's **external secret manager** | Materialized into the instance by the secret provider; read at point of use, never persisted by the platform |

The dividing line is ownership. A secret's custody belongs to the customer's secret manager, so the platform pulls it and keeps no copy. A per-workspace flag has no external owner — an operator sets it in Admin — so the platform is its system of record and stores it.

## Secrets come from the provider, not from us

Secrets are **pulled from the external secret manager**, not stored in the platform database. The secret provider materializes current secret values into the instance (the External Secrets pattern, with SOPS for encrypted-at-rest material), and Configuration & Secrets reads them from there at the point they are used. Nothing secret lands in the platform's own tables, backups, or caches — rotation happens in the manager and flows through automatically, and the customer keeps full custody.

## Portable across customer environments

The platform deploys self-contained into each customer's environment and binds to **whatever secret and config providers that environment already runs** — AWS Secrets Manager, Azure Key Vault, GCP Secret Manager, or a self-hosted OpenBao for secrets; env and ConfigMaps for static config. The binding is configuration: a deployment points the resolver at the customer's providers, and the rest of the platform reads the same way everywhere.

## Layered resolution

For the values the platform does own, resolution is layered: a **workspace-specific value overrides the platform-wide default**. A deployment sets sensible defaults once, and each workspace overrides only what it needs — so most workspaces carry almost no configuration, and every difference is explicit and small.

## Runtime changes take effect live

A change to a platform-owned setting takes effect at runtime: the stored value updates and the short-lived cache invalidates, so a flag flip propagates within seconds — no redeploy, no restart. Secret rotations flow the same way, driven from the manager through the provider. This is what lets Admin and governance retune a live workspace safely.

## Scope

Configuration & Secrets serves **platform-level** settings and credentials. Each domain keeps its own configuration with its owner — playbook taxonomy travels inside the playbook, authorize/deny rules live with the [Policy Gate](/engineering/architecture#band-1--kernel), and case data lives in AI Memory — so this component stays a clean, general resolver rather than a catch-all.

## Dependencies

Uses [Data Adapter](/engineering/architecture/components/data-adapter) (Tier 0) to store and cache the platform-owned settings, and emits telemetry to the cross-cutting [Observability](/engineering/architecture/components/observability). The secret-manager binding runs out-of-band through the secret provider, which keeps the dependency graph clean. It runs as an in-process library, so resolving a value is fast on the hot path.
