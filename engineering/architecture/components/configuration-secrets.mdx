Configuration & Secrets is the single place a component asks for a workspace setting or a credential. It serves general platform configuration — feature flags, operational toggles, connector credentials, and signing keys. Domain configuration lives with its owners: playbook taxonomy with the playbook, authorize/deny rules with the Policy Gate, and case data in AI Memory.

## Contract

| Method | What it does |
| --- | --- |
| `get` | Reads a setting, flag, or secret for the current workspace (an `isSecret` flag marks secrets so they are handled safely) |
| `set` | Creates or updates a setting, flag, or secret (used by Admin and governance) |

## State & storage

Owns the per-workspace settings and feature flags, plus secrets encrypted at rest. A workspace value overrides the platform-wide default. The external secret manager is the source of truth for secrets; Configuration & Secrets keeps a synced, encrypted copy so reads stay fast and local, and keeps secret values out of logs.

## Dependencies

Builds on [Data Access](/engineering/architecture/components/data-access) (Tier 0) and the cross-cutting [Context & Observability](/engineering/architecture/components/context-observability). The secret-manager sync runs out-of-band, which keeps the dependency graph clean.

## Infra & deployment

A relational DB (via Data Access) for settings and encrypted secrets, a cache for hot config reads, and an external secret manager as the source of truth. In-process library; config changes take effect at runtime with short-lived cache invalidation, so updates propagate without a redeploy.
