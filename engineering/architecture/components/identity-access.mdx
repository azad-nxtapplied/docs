<Note>
  **Tier 2 · Foundation** — the top of the foundation. Part of the [NXT Core architecture](/engineering/architecture).
</Note>

Identity & Access is the platform's IAM — one bounded context that answers who is this, what do they belong to, and what may they do. The role assignment lives on membership, so authorization is an internal lookup. It handles coarse capability checks; the Policy Gate handles per-action decisions at execution time.

## Contract

| Method | What it does |
| --- | --- |
| `authenticate` | Verifies a credential (password, API key, service/agent token) and identifies the caller |
| `verify` | Validates an active session or token → who they are, which workspace, and their role |
| `issue / revoke` | Mints or invalidates sessions, tokens, and API keys |
| `membership` | Resolves which workspaces a principal belongs to (and who belongs to a workspace) |
| `can` | Answers whether a principal may use a capability in a workspace → allow / deny |

## State & storage

Owns the account records (users, service/agent identities, sessions, API keys), the tenancy records (orgs, workspaces, memberships, and the role each membership carries), and the role→permission map. Credentials are stored hashed and secured. Identity & Access **defines the workspace**: the `workspace_id` that travels with each request, and that the lower tiers use for tenant scoping, is resolved here — which is why it sits at the top of the foundation.

## Dependencies

Builds on [Data Access](/engineering/architecture/components/data-access) (Tier 0); reads token-signing keys and IdP config from [Configuration & Secrets](/engineering/architecture/components/configuration-secrets) (Tier 1); publishes identity events via [Eventing](/engineering/architecture/components/eventing) (Tier 1); and emits telemetry to the cross-cutting [Observability](/engineering/architecture/components/observability). For federated (OAuth) login, provider keys are synced in and the interactive redirect and token-exchange ride the Integration Fabric.

<Note>
  **Identity & Access and the Policy Gate are complementary.** IAM gates _capability_ (do you have a licence); the Policy Gate gates the _specific action_ at execution time. Together they answer both questions.
</Note>

## Infra & deployment

A relational DB (accounts, tenancy, roles) via Data Access, and a cache for hot session/token lookups, the revocation list, and permission checks, plus an external IdP for federated login. In-process library — `verify` and `can` run on every request's hot path, so they stay fast and local.
