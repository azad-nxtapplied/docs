</Note>

Data Access provides the shared, safe way for every component to reach stateful backends. It owns the connection plumbing for two backends — the **DB** (the durable system of record) and the **cache** (ephemeral storage, locks, and rate buckets) — while each component owns its own tables and keyspace.

## Contract

| Method | What it does |
| --- | --- |
| `get_session` | Returns a safe, tenant-scoped handle to the database for this request |
| `unit_of_work` | Runs several writes as one all-or-nothing transaction |
| `get_cache` | Returns a tenant-scoped handle to the fast, temporary cache |
| `lock` | Holds a short-lived lock so a single worker touches a resource at a time |
| `run_migrations` | Applies pending schema changes |
| `health` | Reports whether the database and cache are reachable |

## State & storage

Its only persistent state is the migration history — the record of which schema changes have been applied. Every request is automatically scoped to its workspace, and the database stays the source of truth while the cache serves as a fast, ephemeral copy.

## Dependencies

Sits at the base of the platform with no internal dependencies; it emits telemetry to the cross-cutting [Observability](/engineering/architecture/components/observability).

## Infra & deployment

A relational DB, a cache/key-value store, and a migration runner. Runs in-process inside the API and workers — it is on every hot path — with one shared connection pool per process.
