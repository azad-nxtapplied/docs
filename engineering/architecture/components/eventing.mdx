Eventing is the platform's internal event backbone. When a component changes state, it publishes an event so the rest of the platform can react — and the event is delivered only once the change commits. It carries events between components inside the platform; external delivery rides the Integration Fabric, and the durable record of decisions lives in the AI Auditor.

## Contract

| Method | What it does |
| --- | --- |
| `publish` | Emits an event as part of the current change; delivered once that change commits |
| `subscribe` | Registers to receive events of a type, with automatic retry and de-duplication |

## State & storage

Owns the **outbox** — events waiting to be delivered — and the bookkeeping of what has published. Because each event is written in the same transaction as the state change, it publishes exactly when that change commits, so events stay consistent with state. Every event carries the request's workspace and correlation id.

## Dependencies

Builds on [Data Access](/engineering/architecture/components/data-access) (Tier 0) and emits telemetry to the cross-cutting [Observability](/engineering/architecture/components/observability). Eventing transports events; consumers (the AI Auditor, evaluation jobs) own what they mean.

## Infra & deployment

An event bus / durable message stream plus the outbox table (via Data Access). `publish` is an in-process library so it commits inside the caller's transaction, and a small background relay drains the outbox and handles retries.
