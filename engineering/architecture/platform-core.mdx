Platform Core is the **Foundation Plane** of [Band 1 · Kernel](/engineering/architecture/nxt-core). It is laid out as **layered tiers** — each tier depends only on tiers below it, never sideways or up. Two concerns are **cross-cutting** (ambient, not tiered).

| Tier | Component | Contract (high level) |
| --- | --- | --- |
| 0 | **Data Access** | `get_session` · `unit_of_work` · `get_cache` · `lock` · `run_migrations` · `health` |
| 1 | **Configuration & Secrets** | `get` · `set` |
| 1 | **Eventing** | `publish` · `subscribe` |
| 2 | **Identity & Access** | `authenticate` · `verify` · `issue/revoke` · `membership` · `can` |
| — | **Context & Observability** | ambient request scope + ops telemetry (cross-cutting) |

<Note>
  Lower tiers never call up to Identity/Tenancy — they read `workspace_id` from the ambient **Context** instead, which keeps the dependency graph acyclic.
</Note>

## Contracts

<AccordionGroup>
  <Accordion title="Tier 0 · Data Access" icon="database">
    The shared, safe way to talk to stateful backends. **NOT a datastore** — every component owns its own tables/keyspace; this owns the connection plumbing for two backends: the **DB** (durable system of record) and the **cache** (ephemeral cache / locks / rate buckets).

    **Contract**

    - `get_session` — a safe, tenant-scoped handle to the database for this request
    - `unit_of_work` — run several writes as one all-or-nothing transaction
    - `get_cache` — a tenant-scoped handle to the fast, temporary cache
    - `lock` — hold a short-lived lock so only one worker touches a resource at a time
    - `run_migrations` — apply pending schema changes
    - `health` — report whether the database and cache are reachable

    **State & storage** — Plumbing, holds no business data; the only thing it persists is which schema changes have been applied. Every request is auto-scoped to its workspace (the cache follows the same rule and is never the source of truth).

    **Dependencies** — Sits at the very bottom; depends on nothing above it, only cross-cutting Context and Observability.

    **Infra & deployment** — A relational DB + a cache/key-value store + a migration runner. Runs in-process inside the API and workers (it is on every hot path); one shared connection pool per process.
  </Accordion>

  <Accordion title="Tier 1 · Configuration & Secrets" icon="sliders">
    The single place a component asks what the setting is for this workspace and gives me this credential. **General platform config only** — feature flags, operational toggles, connector creds, signing keys. **Not** business/domain config (playbook taxonomy lives with the playbook, authorize/deny rules with the Policy Gate, case data in AI Memory).

    **Contract**

    - `get` — read a setting, flag, or secret for the current workspace (an `isSecret` flag marks secrets for safe handling)
    - `set` — create or update a setting, flag, or secret (used by Admin / governance)

    **State & storage** — Owns per-workspace settings + feature flags, plus secrets kept encrypted at rest. A workspace value overrides a platform-wide default. Source of truth for secrets is an external secret manager; a synced encrypted copy keeps reads fast. Secret values are never logged.

    **Dependencies** — Reads/writes through Data Access (Tier 0), plus Context and Observability. The secret-manager sync runs out-of-band — not a runtime dependency, so this stays acyclic.

    **Infra & deployment** — Relational DB (via Data Access) for settings + encrypted secrets, a cache for hot config reads, an external secret manager as source of truth. In-process library; config changes take effect at runtime with short-lived cache invalidation — no redeploy.
  </Accordion>

  <Accordion title="Tier 1 · Eventing" icon="tower-broadcast">
    The platform's **internal event backbone**. When a component changes state it emits an event so the rest of the platform can react — delivered only if the change actually committed. Internal bus only; **not** external I/O (that's the Integration Fabric) and **not** the long-term record (that's the AI Auditor).

    **Contract**

    - `publish` — emit an event as part of the current change; delivered only after that change commits
    - `subscribe` — register to receive events of a type, with automatic retry and de-duplication

    **State & storage** — Owns the **outbox** (events waiting to be delivered) + publish bookkeeping. The guarantee: an event is written in the same transaction as the state change, so it publishes only if that change committed — no lost or phantom events. Every event carries its workspace + correlation id from Context.

    **Dependencies** — Writes the outbox through Data Access (Tier 0), plus Context and Observability. It only transports events — consumers (AI Auditor, eval jobs) own what they mean.

    **Infra & deployment** — An event bus / durable message stream + the outbox table (via Data Access). `publish` is an in-process library so it commits inside the caller's transaction; a small background relay drains the outbox and handles retries.
  </Accordion>

  <Accordion title="Tier 2 · Identity & Access (IAM)" icon="id-card">
    The platform's **IAM** — one bounded context answering who is this, what do they belong to, and what may they do. Merged because the role assignment already lives on membership, so authorization is an internal lookup, not a cross-boundary call. **Coarse capability only**; the per-action autonomy gate is the Policy Gate.

    **Contract**

    - `authenticate` — verify a credential (password, API key, service/agent token) and identify the caller
    - `verify` — validate an active session/token → who, which workspace, what role
    - `issue / revoke` — mint or invalidate sessions, tokens, API keys
    - `membership` — which workspaces a principal belongs to (and who belongs to a workspace)
    - `can` — may this principal use this capability in this workspace? → allow / deny

    **State & storage** — Owns account records (users, service/agent identities, sessions, API keys), tenancy records (orgs, workspaces, memberships + the role each carries), and the role→permission map. Credentials stored hashed/secured. **It defines the workspace** — the scope Data Access and Configuration read from Context originates here, which is why it sits above them.

    **Dependencies** — Persists via Data Access (Tier 0); gets token-signing keys + IdP config from Configuration & Secrets (Tier 1); emits identity events via Eventing (Tier 1); plus Context + Observability. Federated (OAuth) login: provider keys are synced in, but the interactive redirect/token-exchange rides the Integration Fabric.

    **Infra & deployment** — Relational DB (accounts + tenancy + roles) via Data Access, a cache for hot session/token lookups + revocation list + permission checks, an external IdP for federated login. In-process library — `verify` and `can` run on every request's hot path, so they must be fast and local.
  </Accordion>
</AccordionGroup>
